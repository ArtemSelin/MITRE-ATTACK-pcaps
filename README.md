В БЛИЖАЙШЕЕ ВРЕМЯ ДАННЫЙ РЕПОЗИТОРИЙ БУДЕТ НАПОЛНЕН PCAP ФАЙЛАМИ!!!
Репозиторий представляет собой структурированный банк эталонных профилей сетевого трафика и верифицированных дампов сетевых пакетов (**PCAP**). 

Проект разработан как открытый инструмент для проведения комплексного функционального тестирования, валидации правил детектирования и оценки производительности сетевых и комплексных средств защиты информации (**СЗИ**) в различных эксплуатационных режимах.

---

## 🎯 Назначение репозитория

Каталоги pcap-файлов позволяют решать следующие инженерные и аналитические задачи:
1. **Тестирование эффективности функций безопасности СЗИ:** проверка полноты и качества сигнатурных баз, эвристических, поведенческих и корреляционных модулей.
2. **Валидация правил детектирования:** отладка правил, сигнатур, SIEM-корреляций и поведенческих политик перед их промышленным внедрением.
3. **Нагрузочное тестирование:** использование чистых профилей легитимных протоколов для генерации трафика с целью оценки производительности СЗИ под высокой нагрузкой.

## 📂 Что содержится в репозитории

Архитектура каталогов проекта разделена на четыре независимых технологических блока:

<details>
<summary><b>1. Тестирование функций безопасности (Tactics) — Нажмите, чтобы раскрыть</b></summary>

Раздел построен на базе специализированной хронологической матрицы симуляции тактик нарушителя MITRE ATT&CK.

*   `01_TA0043_Reconnaissance_Разведка` — Активное сканирование, инвентаризация и сбор данных.
*   `03_TA0001_Initial_Access_Получение_начального_доступа` — Эксплуатация уязвимостей внешних сервисов, фишинг.
*   `04_TA0002_Execution_Выполнение` — Трафик выполнения удаленных команд и интерпретаторов (PowerShell, CLI).
*   `05_TA0003_Persistence_Закрепление_в_системе` — Трафик создания устойчивых каналов и учетных записей.
*   `06_TA0004_Privilege_Escalation_Повышение_привилегий` — Сетевые проявления локального и доменного повышения прав.
*   `07_TA0005_Stealth_Скрытность` — Обфускация сетевого трафика, маскировка под легитимные протоколы, стеганография.
*   `08_TA0005_Defense_Impairment_Ослабление_защиты` — Попытки удаленного воздействия на компоненты СЗИ, изменение политик.
*   `09_TA0006_Credential_Access_Доступ_к_учетным_данным` — Брутфорс, атаки на протоколы аутентификации (Kerberos, NTLM).
*   `10_TA0007_Discovery_Исследование_системы` — Сканирование внутренней сети (Lateral Discovery) после закрепления.
*   `11_TA0008_Lateral_Movement_Горизонтальное_перемещение` — Эксплуатация легитимных служб администрирования (RDP, RPC, SSH) для продвижения.
*   `12_TA0009_Collection_Сбор_данных` — Сетевой сбор чувствительной информации с внутренних серверов.
*   `13_TA0011_Command_and_Control_Управление_и_контроль` — Сетевой трафик командных центров (C2), обратных оболочек (Reverse Shell).
*   `14_TA0010_Exfiltration_Эксфильтрация_Вывод_данных` — Скрытый и явный вывод информации за периметр (DNS/ICMP-туннелирование).
*   `15_TA0040_Impact_Влияние` — Сетевые маркеры деструктивных действий (DoS/DDoS, деструктивные эксплойты).
</details>

<details>
<summary><b>2. Стандартные сетевые протоколы (Protocols_Standard)</b></summary>

Эталонные дампы легитимного трафика для проверки парсеров СЗИ и генерации базового фона сети:
*   `Network_L3_L4` — Трафик транспортного и сетевого уровней (ICMP, OSPF, BGP, TCP, UDP).
*   `Infrastructure` — Инфраструктурные службы (DNS, DHCP, NTP, SNMP, LDAP).
*   `File_Transfer` — Протоколы передачи данных (FTP, TFTP, SMB v2/v3, NFS).
*   `Web_Protocols` — Веб-трафик традиционных и современных стандартов (HTTP, HTTPS, TLS 1.3, HTTP3/QUIC).
</details>

<details>
<summary><b>3. Промышленные протоколы АСУ ТП / OT (Protocols_Industrial)</b></summary>

Специфичный сетевой трафик технологических сред для валидации глубокого анализа пакетов (DPI) в индустриальных сегментах:
*   `Modbus_TCP` — Базовые команды опроса, чтения и записи регистров.
*   `Siemens_S7` — Трафик сессий оборудования линейки Siemens (S7comm, S7comm-plus).
*   `IEC_60870-5-104` — Телемеханика и управление энергетическими объектами.
*   `DNP3` / `OPC_UA` / `BACnet` — Протоколы автоматизации зданий и сбора технологических данных.
</details>

<details>
<summary><b>4. Контроль приложений (Applications)</b></summary>

Профили трафика популярных пользовательских сервисов для проверки точности работы модулей распознавания приложений (AppID / Application Control):
*   `Messengers` (Telegram, WhatsApp), `Video_Conferencing` (Zoom, Teams), `VPN_Anonymizers` (Tor, OpenVPN, WireGuard, ShadowSocks), `P2P_Torrents` и медиа-стриминг (`Streaming_Media`).
</details>

## 🛡 Что можно тестировать с помощью проекта

| Класс СЗИ / Платформа | Объект и сценарий валидации |
| :--- | :--- |
| **NDR / NTA (Анализ сетевого трафика)** | Проверка алгоритмов поведенческого анализа (модели машинного обучения, базовые линии сети), обнаружение скрытых туннелей, техник Password Spraying, сканирования и горизонтального перемещения внутри сети. |
| **NGFW / Межсетевые экраны** | Точность работы сигнатурного движка IPS, инспекция и разбор SSL/TLS сессий, корректность работы модулей контроля приложений (AppID) и классификации протоколов. |
| **SIEM-системы** | Проверка правил кросс-продуктовой корреляции. Путем циклического воспроизведения дампов (Replay) тестируется полнота сбора сетевых событий, парсинг системных логов, отправляемых сетевыми СЗИ, и скорость генерации инцидентов безопасности. |
| **EDR / XDR системы** | Валидация хостового обнаружения в связке с сетевым контекстом. Проверяется, как конечные агенты реагируют на сетевую активность: например, фиксирует ли EDR сетевое соединение от недоверенного процесса (скрипта PowerShell, вредоносного бинаря C2) в момент воспроизведения PCAP. |
| **Антивирусы / EPP (Защита хостов)** | Тестирование модулей сетевой защиты (Network Attack Blocker / встроенный файрвол). Проверка способности антивируса блокировать эксплойты на сетевом уровне или детектировать передачу вредоносных файлов, передаваемых в незашифрованном виде (например, по HTTP/FTP/SMB). |
| **IDS / IPS (Сетевой уровень)** | Базовая оценка качества сигнатурных баз при распознавании инструментария и известных уязвимостей (CVE). |
| **WAF (Веб-файрволы)** | Защита веб-ресурсов от автоматизированного перебора директорий (Wordlist Scanning), эксплуатации уязвимостей (OWASP Top 10) и веб-краулеров (Web Scraping). |
| **Песочницы (Sandbox) и Почтовые шлюзы** | Валидация разбора почтового трафика (SMTP/IMAP) на этапе доставки фишинга и проверка способности песочниц корректно обрабатывать сетевые запросы вредоносных вложений при их детонации. |

## 🛠 Требования к загружаемым PCAP-файлам

Для поддержания высокого аналитического качества все файлы в репозитории должны строго отвечать следующим критериям:

1. **Анонимизация (Sanitization):** полное отсутствие реальных чувствительных данных. Вся полезная нагрузка (Payload), пароли, сессионные токены, внутренние доменные имена, а также публичные IP и MAC-адреса реальных инфраструктур должны быть заменены на нейтральные (рекомендуется использовать `tcprewrite`, `tshark` или `bittwiste`).
2. **Минимизация и очистка от шума:** дамп должен содержать исключительно те пакеты, которые необходимы для демонстрации конкретной техники или протокола. Весь сторонний сетевой контекст (NetBIOS, ARP-запросы ОС, широковещательный трафик, сторонние TCP-сессии) должен быть отфильтрован.
3. **Строгое именование:** названия файлов должны однозначно идентифицировать содержимое по шаблону:  
   `[ID_техники]_[используемый_инструмент]_[режим/особенность].pcap`  
   *Пример:* `T1595.002_nmap_syn_stealth_scan.pcap`.
4. **Локальное документирование:** каждый конечный каталог обязательно снабжается локальным файлом `README.md`, содержащим описание условий записи трафика: версии использованного ПО, ключи запуска утилит, карту IP-адресов стенда и ожидаемый результат детектирования.
